## React コンポーネント設計方針

- **Props は呼び出し先の関心で設計する**: データの有無（未ログイン・ローディング中等）は呼び出し元の関心事。コンポーネント側では optional にせず必須 props として受け取り、呼び出し元でデフォルト値や表示制御を行う。

## React コーディングルール

- **JSX部にロジックを入れ込まない**: 変数宣言・条件分岐・データ変換などのロジックはJSXの外で完結させ、JSX部は純粋にUIの構造を表現することに徹する。
- **三項演算子をネストしない**: JSX内の条件分岐は `&&` や単一の三項演算子まで。ネストが必要な場合は変数に切り出すか早期リターンで対応する。
- **イベントハンドラはJSXインラインで定義しない**: `onClick={() => doSomething(id)}` のような無名関数をJSX内に書かず、名前付き関数として事前に宣言する。
- **カスタムフックでロジックを分離する**: コンポーネントが肥大化したら、状態管理・副作用をカスタムフックに切り出す。コンポーネント本体は「フックを呼んでJSXを返すだけ」が理想形。
- **1つのカスタムフックは1つの関心事のみを扱う**: 複数の関心事が混在するフックは分割する。フック名から責務が明確に読み取れること。
- **useCallback / useMemo は多用しない**: パフォーマンス上の問題が計測できる高負荷な処理にのみ適用する。推測による早すぎる最適化は避ける。
- **useEffect にはコードコメントを必須とする**: なぜその副作用が必要なのか、何をトリガーに発火するのかをコメントで明示する。

## Pull Request 作成

- PRを作成するときは、リポジトリに `.github/PULL_REQUEST_TEMPLATE.md`（または `.github/pull_request_template.md`、`docs/pull_request_template.md` 等のテンプレート）が存在するか必ず確認すること。
- テンプレートが存在する場合は、その構造・見出し・チェックリストを完全に踏襲してPR本文を作成する。独自のフォーマット（"## Summary" / "## Test plan" 等）で上書きしない。
- このルールはスキル（`/create-pr` 等）経由でも、ユーザーからの直接指示でも同様に適用する。スキルが独自テンプレートを持っていても、リポジトリのテンプレートが優先。
- テンプレートが存在しない場合のみ、デフォルトのフォーマットを使ってよい。

## チケット管理ツールとの同期

- Linear / Notion 等のチケットを起点にタスクを開始した場合、作業中にチケット記載の前提・スコープ・TODO がコードの実態と乖離していないか随時確認すること。
- 乖離を見つけたら、その場でチケット本文やコメントの更新案を提案する（新しく判明した事実、スコープ変更、完了した項目、追加で必要になった作業など）。
- ユーザーが明示的に更新を指示していなくても、チケットベースのタスクでは更新提案は積極的に行ってよい。実際の書き込みはユーザー承認を得てから実施する。

## 外部ドキュメンテーションツール（Linear, Notion 等）での記法

- **テーブル記法は非推奨**: Markdown のテーブル記法はツールごとにレンダリングの互換性が低く、崩れやすい。原則として箇条書き・見出し・太字などシンプルな記法を優先し、テーブルはどうしても表形式が適切な場合にのみ使う。
- **構造はフラットに保つ**: 深いネストや複雑なレイアウトを避け、誰が見ても読みやすいプレーンな構成を心がける。

## コーディング原則

- **複雑さを避けてシンプルさを優先する**: 抽象化・間接参照・汎用化は、明確な必要性があるまで導入しない。読んですぐ理解できるコードが最良のコード。
- **純粋関数の比率を最大化する**: 副作用（I/O、状態変更、外部依存）はシステムの境界に押し出し、ビジネスロジックは入力から出力を返すだけの純粋関数として実装する。
- **デメテルの法則を守る**: オブジェクトの内部構造を連鎖的に辿らない。直接の協調者とだけやり取りし、結合度を低く保つ。
- **イミュータビリティを優先する**: デフォルトで不変とし、可変状態は明確な理由がある場合のみ使う。状態変更が少ないほどバグも少ない。
- **早期リターンでネストを減らす**: ガード節で異常系を先に弾き、正常系のロジックをフラットに保つ。
- **命名に時間をかける**: 良い名前はコメントより雄弁。振る舞いや意図が名前から読み取れるコードを目指す。
