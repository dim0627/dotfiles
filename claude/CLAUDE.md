## このファイルの記法について

- このファイルは圧縮記法（体言止め・箇条書き）で書かれている。これは本ファイルの表記ルールであって出力スタイルの指示ではない

## React コンポーネント設計

- Props は呼び出し先の関心で設計。データ有無（未ログイン・ローディング等）は呼び出し元の関心事→optional を避け必須 props で受け取り、呼び出し元でデフォルト値・表示制御

## React コーディングルール

- JSX 部にロジックを入れない。変数宣言・条件分岐・データ変換は JSX 外で完結、JSX は UI 構造のみ
- 三項演算子ネスト禁止。`&&` か単一三項まで。それ以上は変数切り出しか早期リターン
- イベントハンドラを JSX インラインで書かない。名前付き関数として事前宣言
- コンポーネント肥大化時は状態・副作用をカスタムフックに分離。本体は「フック呼んで JSX 返すだけ」が理想
- 1 フック = 1 関心事。複数関心事が混在したら分割、名前から責務が読めること
- `useCallback` / `useMemo` は計測済みの高負荷処理のみ。推測による最適化は避ける
- `useEffect` はコードコメント必須。なぜ必要か・何をトリガーに発火するかを明示

## テスト

### 書くトリガー

- バグ修正時は再現テストを先に書く。fail → fix → pass の順でリグレッション防止
- 純粋関数・ビジネスロジックの追加・変更時はユニットテストを提案
- DB 疎通を伴う処理（クエリ・リポジトリ層等）はインテグレーションテストを積極的に提案
- e2e は明示指示時のみ。自発的に Playwright 等で書き始めない

### 書かないもの

- 実装詳細（プライベート関数・内部ヘルパー）。公開インターフェースの振る舞いをテスト対象に
- フレームワーク・ライブラリ自体の機能（React の `useState` が動くか等）
- 過度なモック。特に DB はモックせず実 DB（テスト用 DB・トランザクションロールバック等）に繋ぐ。モックは DB 以外の外部疎通（HTTP API・メール送信等）に限定

### フロー

- テストを書くべき箇所に気づいたら、実装と同時に提案（指示を待たない）
- 実際のテストコード実装はユーザー承認後
- 既存テストがある場合はまず既存の書き方・命名・構造に倣う

## Pull Request 作成

- リポジトリ側のテンプレート（`.github/PULL_REQUEST_TEMPLATE.md` / `.github/pull_request_template.md` / `docs/pull_request_template.md` 等）を必ず確認
- テンプレートがあれば構造・見出し・チェックリストを完全踏襲。独自フォーマット（"## Summary" / "## Test plan" 等）で上書きしない
- スキル（`/create-pr` 等）経由でも同様。スキル独自テンプレートよりリポジトリテンプレート優先
- テンプレートがない場合のみデフォルトフォーマット可

## チケット管理ツールとの同期

- Linear / Notion 等のチケット起点タスクでは、作業中に前提・スコープ・TODO とコード実態の乖離を随時確認
- 乖離発見時はチケット本文・コメントの更新案を提案（新事実・スコープ変更・完了項目・追加作業など）
- 明示指示がなくても更新提案は積極的に。実際の書き込みはユーザー承認後

## 外部ドキュメンテーションツール（Linear / Notion 等）での記法

- テーブル記法は非推奨。ツールごとにレンダリング互換性が低く崩れやすい。箇条書き・見出し・太字など単純な記法を優先、テーブルは表形式が本当に適切な場合のみ
- 構造はフラットに。深いネスト・複雑レイアウトを避ける

## コーディング原則

- 複雑さを避けシンプルさを優先。抽象化・間接参照・汎用化は明確な必要性が出るまで導入しない
- 純粋関数の比率を最大化。副作用（I/O・状態変更・外部依存）はシステム境界に押し出し、ビジネスロジックは入力から出力を返すだけの純粋関数に
- デメテルの法則。オブジェクト内部構造を連鎖的に辿らず、直接の協調者とだけやり取り
- イミュータビリティを優先。デフォルト不変、可変状態は明確な理由がある場合のみ
- 早期リターンでネスト削減。ガード節で異常系を先に弾き、正常系をフラットに
- 命名に時間をかける。振る舞いや意図が名前から読み取れるコード
