実装者のコンテキストから切り離された別セッションで、現在のブランチに対する self-review を実行し、結果を GitHub PR に記録した上で、ユーザーがどの指摘に対応するか判断しながら修正まで行うためのスキル。

毎回手動で「`/review` → GH に記録 → 修正対象を選別 → 修正」と指示している一連の流れを 1 コマンドにまとめたもの。

## 前提

- 現在のブランチに対応する Pull Request がすでに存在していること
- このスキルは「別セッション」で実行される想定（実装した本人とは別のコンテキストでレビューすることでバイアスを抑える）
- 対応する指摘の選別は人間が行う（Human-in-the-loop）。スキルが自動で全件修正に入ることはない

## 手順

### 1. 対象 PR の確認

引数の有無に応じて PR 情報を取得する。

- **引数あり（PR URL または番号）**:
  ```bash
  gh pr view <引数> --json number,title,url,state,baseRefName,headRefName
  ```
- **引数なし**: 現在のブランチに紐づく PR を取得
  ```bash
  gh pr view --json number,title,url,state,baseRefName,headRefName
  ```

PR が見つからない場合は、`/create-pr` を促してから終了する。

PR URL からリポジトリの `owner/repo` を特定しておく（後続の `gh api` 等で使用）。

### 2. 公式 /review の実行

Skill ツールで公式の `review` スキルを呼び出し、現在のブランチに対するレビューを実行させる。レビューの姿勢と出力フォーマットを揃えるため、以下の指示も同時に伝える。

```
Skill(skill="review")
```

レビュー姿勢の指示（公式 `/review` 実行時に伝えるプロンプト）:

- 批判的な視点でレビューする。実装を擁護せず、懐疑的な第三者レビュアーとして問題を探しに行く
- 「動くか」だけでなく以下を積極的に疑う:
  - エッジケース・境界値（空入力・null・0 件・巨大入力・並行実行）
  - エラーハンドリングの抜け（失敗パスが握りつぶされていないか）
  - セキュリティ（入力検証・インジェクション・秘匿情報の露出）
  - 既存コードとの不整合・重複（既存の仕組みで済むものを再発明していないか）
- 褒めポイントの列挙は不要。指摘ゼロの場合のみ「問題なし」と簡潔に報告する
- ただし確証のない指摘を断定調で書かない。疑いレベルの指摘は「〜の可能性がある」と明示し、重要度接頭辞を `[imo]` 以下に抑える

整形指示（同時に伝えるプロンプト）:

- 各指摘の冒頭に重要度接頭辞を付ける:
  - `[must]` — バグ・セキュリティ・明確な不具合など、必ず修正すべきもの
  - `[imo]` — 設計・可読性・代替案など、検討の余地があるもの
  - `[nits]` — 些細な指摘（命名・スタイル等）
  - `[fyi]` — 参考情報のみ。対応不要
- 各指摘にファイルパス・行番号を併記する
- 全体サマリは最後にまとめる
- 公式 `/review` 側はフォーマット・レビュー姿勢いずれの指示も柔軟に効くので、上記の方針を明示的に伝えれば従ってくれる

### 3. GitHub への投稿

`/review` の出力を、ユーザーに確認した上で `gh pr comment` で PR にコメント投稿する。

確認は必ず次の 2 段階で行う:

1. **ドラフト全文の提示**: 投稿予定のコメント本文の全文を、通常のテキスト応答としてそのまま（コードブロックで）提示する。`/review` の出力がツール結果やサブエージェント経由で返ってきた場合、ユーザーの画面には表示されていないため、「ユーザーはすでに内容を見ている」と想定してはならない
2. **投稿可否の確認**: 全文を提示した後で「この内容で PR に投稿していい？」と確認する。本文を提示せずに `AskUserQuestion` で確認だけ行うのは禁止

承認後に投稿する:

```bash
gh pr comment {PR番号} --body "$(cat <<'EOF'
（/review の出力をそのまま貼る）
EOF
)"
```

注意:

- 投稿は PR 全体への issue comment（行紐付けではない）。公式 `/review` の出力は本文中にファイルパス・行番号を埋め込む形式なので、これで十分
- 投稿後、コメントの URL を報告する

### 4. 対応対象の選別（HITL）

投稿した指摘を 1 件ずつユーザーに提示し、対応するか判断してもらう。

提示時の情報:

- 指摘の接頭辞（`[must]` / `[imo]` / `[nits]` / `[fyi]`）
- ファイルパス・行番号
- 指摘内容の要約（冗長な場合は端折ってよい）
- レビュアーが示した修正案があればそれも

ユーザーへの問いかけは `AskUserQuestion` を使い、各指摘について以下の選択肢を提示する:

- **対応する**: この指摘を修正対象に含める
- **スキップ**: 今回は対応しない（理由をユーザーが添えてもよい）
- **後で判断**: 一旦保留にし、最後にまとめて再確認

複数件をまとめて 1 つの質問にしてもよいが、`AskUserQuestion` は 1 質問あたり 4 件までなので、件数が多い場合は分割する。`[fyi]` は基本的にスキップ前提で確認のみで構わない。

ユーザーが対応見送りを選んだ指摘は、その旨を最終報告に含めるためにメモしておく。

### 5. 修正の実行

「対応する」と選ばれた指摘について、`[must]` → `[imo]` → `[nits]` の優先度順で修正を実施する。

各指摘ごとに:

1. 該当ファイルを Read で確認
2. レビュー本文の文脈を踏まえて修正
3. 修正内容を簡潔に報告（何を・なぜ変えたか）

ユーザーが修正案に同意しない・別案を出した場合は、その指示に従う。

### 6. コミット・プッシュ

修正が一段落したところで、コミット・プッシュするかユーザーに確認する。承認された場合は `/commit-and-push` を呼んでもよい。

### 7. 完了報告

最終的に以下を報告する:

- 投稿したコメントの URL
- 対応した指摘の件数（`[must]` / `[imo]` / `[nits]` の内訳）
- スキップした指摘とその理由（あれば）
- コミット・プッシュの状態
